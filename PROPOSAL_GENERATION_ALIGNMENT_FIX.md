# Proposal Generation Alignment Fix

## Scope

This fix improves severe-service control valve proposal quality for uploaded RFPs such as LNG compressor recycle / anti-surge packages. It does not redesign unrelated workflows.

## Template Inference Rules

The proposal generator now inspects parsed RFP title, summary, requirements, line items, process conditions, standards, engineering review points, and application keywords.

High-confidence severe-service control valve signals include:

- `severe-service`, `control valve`, `trim`, `actuator`
- `compressor recycle`, `anti-surge`, `LNG`, `fast response`
- `cavitation`, `flashing`, `high differential pressure`
- `hydrogen`, `leakage class`, `traceability`
- `steam conditioning`, `desuperheating`, `spray water`
- `ISA 75.01`, `IEC 60534`, `ASME B16.34`, `NACE MR0175`

For the LNG compressor recycle RFP, the backend infers:

- Industry: Severe-Service Control Valves
- Template: Severe-Service Control Valve Proposal
- Application: LNG Compressor Recycle / Anti-Surge

The upload page also shows a suggested template/application after parsing. Users can still override the visible template selection before generation.

## Section Generation Rules

For severe-service control valve proposals, the generator uses a richer proposal structure:

- Executive Summary
- Project Background / Opportunity Context
- Scope of Supply / Line Items
- Process Conditions / Service Conditions
- Technical Approach / Engineering Basis
- Preliminary Engineering Calculation Summary
- Valve Configuration / Trim / Actuator / Accessories
- Compliance Matrix
- Inspection and Testing Plan
- Quality Assurance / QA-QC
- Documentation & Deliverables
- Technical Deviations / Clarifications
- Risk Assessment
- TBE / Technical Evaluation Summary
- Project Timeline & Delivery
- Executive Dashboard Snapshot

The backend post-processes the AI response to ensure these sections are present for severe-service valve proposals. Safety-critical sections include the required proposal-stage engineering disclaimer.

## Diagram Selection Rules

Default visual type selection is now section-aware:

- Executive Summary: value chain
- Project Background: value chain / opportunity summary
- Scope of Supply / Line Items: package architecture
- Process Conditions: process flow
- Technical Approach / Engineering Basis: engineering dependency map
- Preliminary Engineering Calculation Summary: engineering validation workflow
- Valve Configuration / Trim / Accessories: valve/accessory architecture
- Compliance Matrix: compliance flow / matrix
- Inspection and Testing Plan: workflow
- Quality Assurance / QA-QC: workflow
- Documentation & Deliverables: workflow
- Technical Deviations / Clarifications: risk tree
- Risk Assessment: risk tree
- TBE / Technical Evaluation Summary: TBE matrix
- Project Timeline & Delivery: Gantt
- Executive Dashboard Snapshot: KPI dashboard

Manual diagram type override is preserved. Users can regenerate a section diagram using any supported visualization type.

## App View vs Export View

The live proposal page remains a continuous scroll document. It renders each proposal section inline with its selected/default diagram and the active visualization type.

PDF and DOCX export paths remain separate and paginated. TOC and export behavior are preserved.

## Vault Retrieval Improvements

Proposal generation now retrieves Knowledge Vault documents and text entries from all users in the same organization, not only the current user. This matters for demo accounts where engineering knowledge is owned by an Engineering Manager but proposals are generated by the Proposal Director.

Vault context is ranked against severe-service keywords and RFP text so relevant compressor recycle, anti-surge, trim, actuator, inspection/testing, standards, deviation, and documentation content is more likely to be used. Proposal metadata counts vault-sourced sections/documents from the final generated sections.

## Known Limitations

- The database schema stores only `industry` and `templateType`, so user-friendly severe-service metadata is encoded in `templateType` and rendered cleanly in the proposal UI.
- Diagram persistence is still client-side for manually generated live diagrams; export routes generate their own section-aware visuals.
- Diagram-inclusive exports should still be visually spot-checked before a boardroom demo.
- AI parsing quality depends on source document clarity and model response quality; backend post-processing ensures section presence but cannot certify engineering correctness.

## Future Roadmap

- Add explicit database fields for inferred industry label, application/subtype, and inference confidence.
- Persist per-section visualization type and generated diagram code.
- Add a first-class severe-service template to the template library instead of encoding it as inference logic.
- Add automated visual regression tests for live proposal diagrams, PDF, and DOCX exports.
- Add richer rule-based extraction for process conditions and line-item tables before LLM analysis.
